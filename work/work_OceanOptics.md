# Ocean Optics USB2000/USB4000 対応 実装計画

## 背景

当初のユーザー案（1D固定分光器として`seabreeze`経由でUSB2000/4000系を追加する）は方向性として
妥当だが、案が作られた時点から`ConfigurationCatalog`（バージョン管理された較正・設定カタログ、
[work_configuration.md](work_configuration.md)参照）、カメラハードウェア識別クロスチェック
（`identity_ready`シグナル）、カメラ側capability自己申告パターン（`temperature_capability_ready`/
`em_gain_info_ready`）が新たに導入されており、現行コードを読み直した結果、当初案のままでは
以下が起きることが分かった。

- `CameraThread`が実際に持つべきシグナル契約（`init_failed`/`hardware_error`/
  `temperature_capability_ready`/`identity_ready`/`em_gain_info_ready`/`em_gain_set_finished`）
  が当初案の想定より広く、欠けると[ui.py:779-797](../src/ui.py#L779-L797)の`.connect()`で
  `AttributeError`になり起動できない。
- `ConfigurationCatalog`は較正スロットの識別子として`grating_index`/`grating_grooves_per_mm`/
  `target_center_wavelength_nm`/ROIを使っており（[file_io_mixin.py:167-220](../src/ui_mixins/file_io_mixin.py#L167-L220)）、
  `combo_grating`が空だと`_current_grating_definition()`が`ConfigurationError`を送出する
  （[file_io_mixin.py:135-153](../src/ui_mixins/file_io_mixin.py#L135-L153)）。当初案の
  `spectrometerConfig.json`は`"grating"`キーを省略しており、これだと「ネオン較正→設定保存」が
  そのまま壊れる。
- Ocean Opticsは物理的に**1台のUSBデバイス**が分光と検出を兼ねるが、現行アーキテクチャは
  カメラ（`CameraThread`）と分光器（`SpectrometerController`）を独立した別ハードウェアとして
  設計しており、`spec_ctrl.initialize()`は`CameraThread`が構築される**前**にメインスレッドで
  同期的に呼ばれる（[ui.py:709](../src/ui.py#L709) → [ui.py:778](../src/ui.py#L778)）。
  両者が別々に`seabreeze`ハンドルを開こうとすると衝突しうる。当初案はこの点に触れていない。
- 分光器側capability（grating/center可動の有無）を静的JSONで表現する仕組みは、現行コードの
  どこにも存在しない。既存パターンは全てハードウェア接続後の自己申告（シグナル）である。

本ドキュメントはこれらを踏まえて実装方針を確定し、Step単位の実行計画としてまとめる。

## 採用する方針

### 方針1: CameraThreadの現行シグナル契約に全面準拠する

`CameraThreadOceanOptics`は[ui.py:779-797](../src/ui.py#L779-L797)が無条件で`.connect()`する
以下を**必ず**実装する（値が無い項目も「無い」ことを報告するシグナルを出す）。

```
data_ready(str, np.ndarray)          # ("1d", intensities) のみ使用
init_finished()
init_failed(str)
hardware_error(str)
temperature_ready(float, str)        # 冷却なし: 呼ばれたら (-1.0, "unsupported") 等を返す
temperature_capability_ready(bool, bool, float, float)  # 常に (False, False, 0.0, 0.0)
acquisition_failed(str)
em_gain_info_ready(bool, bool, int, int, int, int)      # 常に (False, False, 0, 0, 0, 0)
identity_ready(str, str)             # (model, serial_number)
exposure_set_finished()
em_gain_set_finished(int)            # 使われないが呼び出し先が存在する前提のため実装
temperature_set_finished(float)      # 同上
```

`status_ready`/`exposure_applied`/`roi_applied`は[ui.py:788-793](../src/ui.py#L788-L793)で
`hasattr`判定されるためオプション。`status_ready`（Check Camera Status ダイアログ用）は
実装を推奨（P1）、`exposure_applied`/`roi_applied`はAndor固有の詳細読み戻し機構なので
実装しない。

**根拠:** これらのシグナルは`camera_princeton.py`が実証済みの最小実装パターン
（`_report_em_gain_capability`/`_report_temperature_capability`、存在しない機能は
`exists=False`を返して該当UIを`setVisible(False)`にする）をそのまま流用できるため、
Ocean Optics固有のUI無効化コードを書く必要がない。当初案が想定していなかった
「機能が無いことをGUIに伝える責務」を、既存のexists駆動show/hide機構に乗せることで
実装コストを増やさずに解決する。

### 方針2: カメラと分光器は物理的に同一デバイス — ハードウェアハンドルはCameraThreadのみが保持する

`SpectrometerControllerOceanOptics`は**自身ではUSBデバイスを開かない**。`initialize()`は
実際のUSB接続を試みず、常に成功相当（`is_initialized = True`）を返す純粋なno-opとする。
`get_wavelength()`/`get_grating()`/`get_cached_hardware_metadata()`/`get_device_identity()`は
configの静的値またはプレースホルダーを返すのみで、`seabreeze`には触れない。

実機の識別情報（model/serial_number）とnative wavelength配列は、`CameraThreadOceanOptics`が
接続時に一度だけ読み取り、`identity_ready`と`get_cached_hardware_metadata()`経由でのみ
GUIに渡す。`ConfigurationCatalog`の互換性チェックは`spectrometer_serial_number`と
`camera_serial_number`を独立に比較する（[configuration_catalog.py:521-522](../src/configuration_catalog.py#L521-L522)）ため、
分光器側の識別情報を空のままにすると較正の互換性チェックが常に緩くなってしまう。これを防ぐため、
`on_camera_identity_ready`（[acquisition_mixin.py:311-315](../src/ui_mixins/acquisition_mixin.py#L311-L315)）に
`self.config.get("model") == "OceanOptics"`のときだけ、カメラが報告したのと同じ
model/serial_numberで`check_and_record_hardware_identity("spectrometer", ...)`も呼ぶ分岐を追加する
（Step 4）。

**根拠:** `spec_ctrl.initialize()`は`CameraThread`が構築される前にメインスレッドで同期的に
呼ばれ（[ui.py:709](../src/ui.py#L709), [ui.py:778](../src/ui.py#L778)）、両者の接続タイミングを
調停する仕組みが現行アーキテクチャに無い。`seabreeze`（cseabreeze backend）は同一プロセス内で
同じUSBデバイスに対する二重オープンを想定しておらず、確実に安全な設計は「ハンドルを持つのは
1オブジェクトだけ」に限定することである。この前提は実機/実際の`seabreeze`挙動での確認が
取れていないため、Step 1の実行プロンプトに確認事項として明記する。

### 方針3: ConfigurationCatalogとの整合のため、gratingはダミーの単一エントリを残す

Ocean Optics用`spectrometerConfig.json`から`"grating"`キーを省略せず、常に単一エントリを
持たせる。

```json
{
    "model": "OceanOptics",
    "serial_number": null,
    "seabreeze_backend": "cseabreeze",
    "correct_dark_counts": true,
    "correct_nonlinearity": true,
    "grating": [
        {"index": 1, "grooves": 0, "defaultROI": {"from": 0, "to": 1}}
    ],
    "flip_x": false,
    "hardware_identity": {
        "spectrometer": {"model": null, "serial_number": null},
        "camera": {"model": null, "serial_number": null}
    }
}
```

`grooves: 0`は「gratingという概念が意味を持たない」ことを表すセンチネル値として扱う
（UI上はStep 5でgrating comboごと隠すため、ユーザーが直接目にすることはない）。
`target_center_wavelength_nm`は`physical_center_wl`の初期値（[ui.py:171](../src/ui.py#L171)の
`694.0`固定を、Ocean Opticsでは接続後にnative wavelengthの中央値へ上書きする）をそのまま
較正スロット識別に使う。

**根拠:** `_current_grating_definition()`（[file_io_mixin.py:135-153](../src/ui_mixins/file_io_mixin.py#L135-L153)）は
`combo_grating`に選択項目が無いと`ConfigurationError`を送出し、これは較正の`register_current_configuration()`
（[file_io_mixin.py:167](../src/ui_mixins/file_io_mixin.py#L167)）を経由する「ネオン較正→保存」フローを
直接壊す。API側の`_api_configuration_state()`は同じ例外をtry/exceptで吸収済み
（[api_mixin.py:357-363](../src/ui_mixins/api_mixin.py#L357-L363)）でAPI経由は無傷だが、GUI側は
ガードされていない。gratingキー自体を無くすより、ダミーエントリを1つ残す方が既存コードへの
変更が最小で済む。

### 方針4: capability報告はハードウェア自己申告に寄せ、静的JSONの`capabilities`辞書は導入しない

当初案の`spectrometerConfig.json`内`"capabilities": {...}`ブロックは採用しない。既存の
温度/EM gain（方針1）はいずれも「接続したハードウェアがシグナルで自己申告する」方式であり、
静的JSONにフラグを置くと設定ファイルと実際のハードウェアの乖離（例: ユーザーがJSONを誤編集）
を検知できない。代わりに`SpectrometerControllerOceanOptics`に次のプロパティ/メソッドを追加し、
GUI初期化シーケンスの中でこれを読んでgrating comboと関連コントロールの表示/有効を決める。

```python
def get_capabilities(self) -> dict:
    return {"supports_grating": False, "supports_movable_center": False}
```

Andor/Princeton側にも同名メソッドを追加し常に`True`/`True`を返すようにする（ダックタイピングの
一貫性を保つため）。既存の`hasattr`パターンに倣い、`hasattr(self.spec_ctrl, "get_capabilities")`
が無い場合は従来通り「両方True」扱いにフォールバックし、既存2機種の挙動は変えない。

なお、2D Image View/vertical ROIの可否はこの`get_capabilities()`には含めない。検出器の行数
（`self.thread.det_height`）という既に存在する情報から直接判定できるため、新しいcapability
信号をカメラ側に追加する必要はない（詳細はStep 5）。分光器の可動性（grating/center）と
検出器の形状は別のハードウェア特性であり、責務を混在させない。

### 方針5: axis_source を拡張し、「native wavelength・未較正」状態を新設する

`get_x_axis()`（[acquisition_mixin.py:48-80](../src/ui_mixins/acquisition_mixin.py#L48-L80)）は
現在`self.calib_coeffs is None`（pixel軸）か否かの二値でしか分岐しない。Ocean Opticsの
「native wavelengthは使えるが、FluoraPressée較正はまだ無い」という第三の状態を追加する。

`_axis_state()`（[measurement_metadata.py:120-144](../src/measurement_metadata.py#L120-L144)）の
`axis_source`許可リストは`"neon_polynomial"`/`"loaded_configuration"`/`"api_inline_calibration"`/
`"hardware_shamrock"`に限定されており、未登録の文字列は黙って`"loaded_configuration"`に丸め
られる。新しい`axis_source = "oceanoptics_native"`をこのリストに追加しないと、保存メタデータの
`axis.source`が誤表示される（Step 6で対応）。

### 方針6: SpectrometerMoveThreadの統一は任意（本計画のブロッカーではない）

当初案は「`src/spectrometer.py`がimport時にconfigを読んでvendor別importしている」ことを
問題視していたが、現行の[spectrometer.py:20-30](../src/spectrometer.py#L20-L30)は**呼び出し時**に
`controller.__class__.__module__`を見て分岐しており、この問題は既に解消済みである
（`spectrometer_princeton.py`/`spectrometer_andor.py`双方が同じ形の`SpectrometerMoveThread`を
個別に持っているだけ）。Ocean Optics用の分岐を1つ追加するだけで良く、リファクタは不要。

**重要（レビュー指摘を反映、P0）:** `spectrometer_andor.py`の`SpectrometerMoveThread.run()`
（[spectrometer_andor.py:252-255](../src/spectrometer_andor.py#L252-L255)）は
`set_grating()`/`set_wavelength()`の戻り値を見ずに`finished_signal.emit()`するだけで、
`self.success`を一切設定しない。GUI側は`getattr(self.spec_move_thread, "success", True)`
（[spectrometer_control_mixin.py:261](../src/ui_mixins/spectrometer_control_mixin.py#L261)）で
`success`属性が無ければ成功扱いにするため、Andorではこれが実質的に問題にならない
（実機の移動はほぼ常に成功するため）。しかしOcean Opticsでは、方針2の設計により
`set_wavelength()`/`set_grating()`が要求値の不一致を検知して意図的に`False`を返す
ことがあるため、**`spectrometer_andor.py`ではなく`spectrometer_princeton.py`の
`SpectrometerMoveThread.run()`（[spectrometer_princeton.py:565-587](../src/spectrometer_princeton.py#L565-L587)）
を実装の参考にする**必要がある。Ocean OpticsのSpectrometerMoveThreadは次を満たす:

```python
def run(self):
    grating_ok = self.spec_ctrl.set_grating(self.grating_index)
    wavelength_ok = self.spec_ctrl.set_wavelength(self.wavelength)
    self.success = grating_ok and wavelength_ok
    if not self.success:
        self.error_message = (
            "Ocean Optics is a fixed spectrometer; the requested grating/centre "
            "wavelength does not match the connected device's fixed position."
        )
    self.finished_signal.emit()
```

これにより、`on_spectrometer_moved`（[spectrometer_control_mixin.py:222-272](../src/ui_mixins/spectrometer_control_mixin.py#L222-L272)）の
既存の失敗経路（`QMessageBox.warning`/`_fail_pending_configuration`）が正しく発火し、
異なるcenterを要求するConfiguration Loadが「移動成功」として誤って較正適用されることを防ぐ。
（`spectrometer_andor.py`自体にこの欠落があることは今回の調査で判明した別件であり、
Andorの実機移動は現状ほぼ常に成功するため実害は小さいが、本計画のスコープ外として
修正は行わない。）

## ステップ間の依存関係

Step 1（`camera_oceanoptics.py`）は他に依存せず最初に着手できる（既存の`CameraThread`契約さえ
守れば、`ConfigurationCatalog`やUI無効化の設計判断を待たずに書ける）。Step 2はStep 1の
identity共有規約（方針2）を前提とするため、Step 1と内容をすり合わせながら実施する。Step 3は
Step 1・2の完成後、数行の分岐追加のみ。Step 4（identity統合・ダミーgrating運用の実地確認）は
Step 1-3が動く状態でないと検証できない。Step 5（capability駆動UI無効化）はStep 2の
`get_capabilities()`に依存する。Step 4の固定center同期とStep 5のUI無効化はどちらも
`on_camera_initialized`内に実装するため、実際にはこの2つを同じコード変更としてまとめて
実施してよい（依存関係上は別Stepとして記載しているが、実装単位を分ける必要はない）。

> （EM gain、温度制御）は`exists=False`相当を報告する。
>
> - `run()`の非debug分岐で`seabreeze`を遅延import（モジュール先頭ではimportせず、`run()`内
>   または接続処理内で行う）。未インストール時は
>   `init_failed.emit("Ocean Optics support requires seabreeze. Install with: pip install "
>   "seabreeze  (or run setup_oceanoptics.bat/.sh)")`で分かりやすく落とす。

>   `import seabreeze`してから`from seabreeze.spectrometers import Spectrometer`する**前**に

>   if backend_name:

>   使用可能に見えてしまう）:

>   2. `np.all(np.isfinite(wavelengths))`（全値がfinite）
>   3. `np.all(np.diff(wavelengths) > 0)`（厳密に単調増加）
>   検証に失敗したら`init_failed.emit("Ocean Optics device reported an invalid wavelength "
>   "calibration array: ...")`を呼び、`run()`をそのまま終了する（`spec.close()`してから
>   return）。検証を通れば`self.det_width = len(wavelengths)`、`self.det_height = 1`とし、

>   `em_gain_info_ready.emit(False, False, 0, 0, 0, 0)`を接続直後に一度emitする。
>   `read_temperature()`が呼ばれても、常に`(0.0, "unsupported")`を`temperature_ready`で
>   返す（**`-999.0`は使わない** — `on_temperature_read()`
>   （[acquisition_mixin.py:242-268](../src/ui_mixins/acquisition_mixin.py#L242-L268)）は
>   `if temp == -999.0`を最初にチェックして無条件で"Error"表示にしてしまい、後続の
>   `status == "unsupported"`分岐（本来使いたい方）まで到達しない。この経路は方針5により
>   温度UI自体が非表示になるため実害はないが、`0.0`を使えば意図通り"unsupported"分岐に
>   乗る）。
> - **排他制御（`camera_princeton.py`と同型で必須）:** `_lock`（`Condition`、`new_exposure`/
>   `is_measuring`等のリクエスト状態用）と`_hw_lock`（`Lock`、`spec`への実アクセス―
>   `intensities()`/`integration_time_micros()`/`wavelengths()`/`close()`―を直列化する専用
>   ロック）を分離して持つ。`update_exposure(exp_time)`は`_exposure_request_seq`を
>   インクリメントして返す（トークン）。メインループは`_hw_lock`配下で
>   `spec.integration_time_micros(int(exp_time * 1e6))`を呼び、成功・失敗いずれでも
>   `_exposure_applied_seq`を更新して`self._lock.notify_all()`し、`exposure_set_finished.emit()`
>   する。`wait_for_exposure_applied(seq, timeout=None)`を実装する
>   （[camera_princeton.py:1177-1186](../src/camera_princeton.py#L1177-L1186)と同型）。
>   **これは`_api_start_acquire()`（[api_mixin.py:228-237](../src/ui_mixins/api_mixin.py#L228-L237)）が
>   `exposure_s`指定時に無条件で`self.thread.update_exposure(...)`の戻り値を渡して
>   `self.thread.wait_for_exposure_applied(seq, timeout=...)`を呼ぶため必須の契約であり、
>   欠けるとAPI経由の露光変更付き取得が`AttributeError`になる。**
> - **dark/nonlinearity補正のcapability確認（必須。2回に分けて試験する — レビュー指摘を反映）:**
>   `config`の要求値をそのまま`intensities()`へ渡すのは危険。python-seabreeze公式実装では、
>   `intensities()`内部が`if correct_dark_counts and not self._dp: raise SeaBreezeError(...)`
>   に続けて`if correct_nonlinearity and not self._nc: raise SeaBreezeError(...)`という
>   **逐次の`if`文**になっているため（dark_countsが先）、`intensities(True, True)`を**1回だけ**
>   試すと、dark_countsが非対応の場合は例外がそこで発生してnonlinearity側のチェックに到達
>   せず、2つの補正の対応可否を独立に判定できない。したがって接続直後に**2回、独立に**試験する:
>   ```python
>   try:
>       spec.intensities(correct_dark_counts=True, correct_nonlinearity=False)
>       supports_dark = True
>   except SeaBreezeError:
>       supports_dark = False
>   try:
>       spec.intensities(correct_dark_counts=False, correct_nonlinearity=True)
>       supports_nonlinearity = True
>   except SeaBreezeError:
>       supports_nonlinearity = False
>   ```
>   さらに`spec.intensities(correct_dark_counts=False, correct_nonlinearity=False)`
>   （無補正の通常取得）も1回試し、これが失敗する場合は取得そのものが不可能な異常事態
>   として`init_failed`にする。判定結果は`self._supports_dark_correction`/
>   `self._supports_nonlinearity_correction`としてキャッシュする。以降は`config`の要求値と
>   実サポートの論理積（`requested and supported`）を使い、要求されたが非対応の場合は
>   接続時に一度だけ警告してフォールバックする（旧機種のUSB2000はnonlinearity補正が
>   非対応の可能性が高く、既定値`true`のままでは対応対象機種で取得自体が失敗しうる）。
>   `get_cached_hardware_metadata()`が返す`hardware_dark_corrected`/`nonlinearity_corrected`は、
>   config要求値ではなく**実際に適用した値**とする（Step 7で背景不一致判定に使うため）。
> - 測定ループは上記で確定した実際の補正フラグを使って`_hw_lock`配下で`spec.intensities(...)`を
>   呼び、`data_ready.emit("1d", intensities)`する。`roi_mode`/`update_roi_settings()`は受理するが
>   実際には何もしない（Step 5でUIから到達不能になるため、来ても無視して良い）。
> - **`close()`の所有権を明確にする（レビュー指摘、`camera_princeton.py`と同一パターンに
>   揃える）:** `spec.close()`を呼ぶのは`run()`の`finally`ブロックのみとする
>   （[camera_princeton.py:1123-1132](../src/camera_princeton.py#L1123-L1132)の
>   `finally: if self.cam is not None: self.cam.close(); self.cam = None`と同型）。
>   `stop_thread()`自身は`self.thread_active = False`を設定して`self.wait()`（`QThread.wait()`、
>   `run()`が実際に返るまでブロック）を呼ぶだけにする
>   （[camera_princeton.py:1301-1303](../src/camera_princeton.py#L1301-L1303)と同型）。
>   こうすることで「`spec`を閉じるのは`run()`を実行しているスレッド自身」という所有権が
>   一箇所に定まり、`_hw_lock`配下で進行中の`intensities()`呼び出しと`close()`が別スレッドから
>   競合する余地がなくなる。
> - `--debug`分岐は`camera_princeton.py`のダミースペクトル生成（ネオン風の二重ピーク+ノイズ、
>   [camera_princeton.py:1010-1023](../src/camera_princeton.py#L1010-L1023)）を流用し、
>   `det_width`はダミーのnative wavelength配列長（例: 2048点、350-1050nm相当の線形配列）に
>   合わせる。`identity_ready.emit("USB4000 [DEBUG]", "DEBUG-OCEANOPTICS-0000000")`のように
>   実機と同じ形で発行する。
>
> **`acquire_single_image()`/`camera`プロパティは実装しない（当初計画からの変更）:**
> リポジトリ全体を`grep`した結果、`camera_princeton.py`/`camera_andor.py`の
> `acquire_single_image()`と`camera`プロパティは**現在どこからも呼ばれていない**
> （`camera_andor.py:577`のdocstringは「`calibration_ui`から呼ばれるプロキシ」と書いて
> いるが、実際の`calibration_ui.py`は`update_exposure()`/`start_measuring()`/`data_ready`
> シグナルによる非同期取得のみを使っており、この記述は古いまま残された死んだコードである）。

**テスト計画（P0、当初計画では手動確認のみだった点を追加）:** `--debug`経路では通らない

## Step 6: Native wavelength軸の統合と警告ラベル

**優先度:** P1
**依存:** Step 1
**対象ファイル:** `src/ui_mixins/acquisition_mixin.py`, `src/measurement_metadata.py`, `src/ui.py`,
`src/ui_mixins/display_mixin.py`, `src/ui_mixins/file_io_mixin.py`

**実行プロンプト:**

> `get_x_axis()`（[acquisition_mixin.py:48-80](../src/ui_mixins/acquisition_mixin.py#L48-L80)）に、
> `self.calib_coeffs is None`かつ`hasattr(self.thread, "native_wavelengths")`
> （Ocean Opticsのみ実装するキャッシュ済み属性、Step 1参照）の場合の分岐を追加し、
> `np.arange(num_pixels)`の代わりに`self.thread.native_wavelengths`を返す。**この配列は
> `get_cached_hardware_metadata()`経由では取得しない**（理由はこの下の「metadata分離」節）。
> Raman modeがONの場合は既存の nm→Raman shift 変換ロジック（同メソッド内、
> `calib_is_raman`/`display_is_raman`分岐と同型の変換）をnative wavelengthに対しても適用する。
>
> **`public_axis_kind()`をここで定義する（Step 8で参照されるため、定義箇所をここに一本化する）:**
> `src/measurement_metadata.py`に
> `def public_axis_kind(window) -> Literal["pixel", "native_wavelength", "calibrated"]`を追加する。
> ロジックは: `calib_coeffs is not None`なら`"calibrated"`、それ以外で
> `hasattr(window.thread, "native_wavelengths")`なら`"native_wavelength"`、どちらでもなければ
> `"pixel"`。`_axis_state()`（[measurement_metadata.py:120-144](../src/measurement_metadata.py#L120-L144)、
> 方針5）にも`axis_source`として`"oceanoptics_native"`を許可リストへ追加し、`get_x_axis()`と
> 同じ条件でこの値を設定する（`_axis_state()`はGUI保存メタデータ向けの詳細な由来を表す語彙で、
> `public_axis_kind()`はAPI/表示ロジック向けの単純化された3値— 両者は別物として維持する）。
>
> **P0: `flip_x`とnative wavelengthの組み合わせ不具合を修正する（レビュー指摘、必須）:**
> 既存の表示・保存コードは「X軸を反転するかどうか」を`self.calib_coeffs is not None`で
> 判定しているが、これは「pixel index（反転不要、`np.arange`は対称なのでY軸だけ反転すれば
> 見た目上ミラーになる）」と「意味のある物理軸（wavelength/Raman shift、X・Y両方を反転しないと
> 対応がずれる）」を区別するためのものであり、`calib_coeffs is None`だが物理的に意味のある
> native wavelength軸を使うOcean Opticsのケースを想定していない。放置すると、`flip_x=True`時に
> 強度配列だけ反転されnative wavelength配列は反転されないため、**波長と強度の対応が丸ごと
> ずれる**。次の3箇所を`self.calib_coeffs is not None`ではなく
> `public_axis_kind(self) != "pixel"`で判定するよう修正する:
> 1. `update_display()`のX軸反転
>    （[display_mixin.py:95](../src/ui_mixins/display_mixin.py#L95)）。
> 2. `on_mouse_moved()`のindex検索方式の分岐
>    （[display_mixin.py:310](../src/ui_mixins/display_mixin.py#L310)）— pixel軸なら
>    `disp_idx = x_pixel`のまま、それ以外（native_wavelength/calibrated）なら既存の
>    `x_arr`最近傍探索を使う。
> 3. `on_mouse_moved()`の座標ラベル分岐
>    （[display_mixin.py:326](../src/ui_mixins/display_mixin.py#L326)）— pixel軸なら
>    従来通り"Pixel: ..."のみ、それ以外ならWavelength/Raman shiftとして表示する（native
>    wavelength時もこの分岐に入るよう修正が必要 — 当初コードはここが`calib_coeffs is not
>    None`のみで、native wavelength時に誤って"Pixel: ..."表示に落ちていた）。
> 4. `_save_data_to_path()`のX軸反転
>    （[file_io_mixin.py:554](../src/ui_mixins/file_io_mixin.py#L554)）。
>    （`raw_data`/`bg_data`の反転はflip_xのみに依存する既存の無条件反転のままで良い —
>    x_dataと対になるy系列は常にflip_xと連動して反転するのが正しい設計であり、
>    バグはx_data自体の反転条件にのみ存在する。）
>
> `update_plot_labels()`（[spectrometer_control_mixin.py:50-60](../src/ui_mixins/spectrometer_control_mixin.py#L50-L60)）
> に、`public_axis_kind(self) == "native_wavelength"`の場合は
> `'Wavelength (nm) [Ocean Optics factory calibration]'`のようなラベルを出す分岐を追加する。
>
> **用語の注意（レビュー指摘を反映）:** `spec.wavelengths()`は生のpixel番号ではなく、機器
> 内蔵の工場較正係数から導出されたnm軸であり、単純な「uncalibrated（未較正）」ではない。
> 「FluoraPressée自身のネオン較正が未適用」であることと「較正が全く無い」ことは区別し、
> ラベル・警告文・API（Step 8）のいずれも"factory-calibrated, not verified against
> FluoraPressée's own calibration"の意味であることが伝わる表現にする（"uncalibrated"は
> 単独では使わない）。
>
> **metadata分離（レビュー指摘を反映、P1）:** `native_wavelengths`（数千点のNumPy配列）を
> `get_cached_hardware_metadata()`の戻り値へそのまま含めない。`_camera_state()`
> （[measurement_metadata.py:80-101](../src/measurement_metadata.py#L80-L101)）はこの戻り値を
> そのまま`state`へマージして測定結果メタデータ・保存ファイルへ書き込むため、大きな配列が
> 全保存ファイルへ混入するだけでなく、`json_value()`（[instrument_status.py:11-28](../src/instrument_status.py#L11-L28)）は
> `ndarray`を`tuple`/`list`/`set`のいずれとも認識せず（`.item()`も要素数1以外では
> `ValueError`でスキップされる）最終的に`str(value)`へフォールバックするため、
> `numpy`の省略記法（例: `"[350.1 350.2 ... 1049.8]"`）がJSON文字列として保存されてしまう
> （実質壊れたデータ）。`get_cached_hardware_metadata()`には代わりに
> `"native_wavelength_range": {"count": int, "min_nm": float, "max_nm": float}`のような
> 要約のみを含める。配列そのものは`self.thread.native_wavelengths`という直接属性としてのみ
> 公開し、`get_x_axis()`はこちらを参照する（`get_cached_hardware_metadata()`経由にしない、
> という上の修正と整合させる）。
>
> `ui.py`にプロットの左上へ表示する警告ラベル（`QLabel`、既存の`coord_label`や
> `lbl_loaded_configuration`と同系統のスタイル）を追加し、Ocean Optics接続中かつ
> `public_axis_kind(self) == "native_wavelength"`のときだけ
> "X-axis: Ocean Optics factory-calibrated wavelength (no FluoraPressée calibration applied)"
> を表示、較正適用後は非表示にする（`apply_calibration()`/`clear_active_configuration()`の
> 呼び出し箇所に表示切り替えを追加）。

**検証:** Ocean Optics・較正未適用でプロットのX軸がnative wavelength（nm）になり、
上記文言の警告ラベルが表示されること。ネオン較正適用後はラベルが消え、従来通りの較正済み
表示になること。**`flip_x`のON/OFF双方について**、表示上のX/Y対応と保存ファイルのX/Y/raw/bg
列の対応が崩れていないことを、native wavelength・pixel・calibrated較正済みの3状態それぞれで
確認するテストを`tests/`に追加する（既知のダミースペクトルに対し、ピークの実際の波長位置が
flip_x ON/OFFで変わらないことを検証する形が分かりやすい）。マウスホバー時の座標ラベルが
native wavelength時に"Pixel: ..."ではなく"Wavelength: ..."になることも確認する。
既存Andor/Princeton（pixel/calibrated）表示の回帰が無いことも確認する。

テストが壊れないこと（`tests/test_baseline_io_api.py`等）。

## Step 8: API `x_axis` メタデータ追加

>

>
> `api_acquire()`（[api_mixin.py:448-546](../src/ui_mixins/api_mixin.py#L448-L546)）の
> レスポンス組み立てに、`public_axis_kind(self)`を参照して

> `source`（および`_api_configuration_state()`の`axis_mode`)は
> `"pixel"`/`"native_wavelength"`/`"calibrated"`の3値に**統一する**（当初案にあった
> `"fluora_calibration"`という別名は使わない — `axis_mode`と`x_axis.source`で異なる
> 文字列を使う理由がないため）。`unit`の優先順位を明記する（レビュー指摘を反映）:
> **`source == "pixel"`なら常に`None`**（Ramanトグルの状態に関わらず — pixel軸に単位は
> 無いため最優先）、それ以外（`"native_wavelength"`/`"calibrated"`）はRaman modeなら
> `"cm-1"`、そうでなければ`"nm"`とする。`calibrated: true`は`source == "calibrated"`と
> 等価（すなわち「FluoraPressée較正が適用済み」）であることをAPIドキュメント
> （`manuals/API.md`）に明記する（Step 6の用語整理と同じ理由）。後方互換性は考慮不要
> （API未運用のため、CLAUDE.mdの既存記載通り）。

   テスト: `tests/test_configuration_api.py`に`test_acquire_response_includes_x_axis`、
   `tests/test_api_oceanoptics_mixin.py`の`ConfigurationUnitLabelTests`(4件)。

3. **P1: ハードウェア情報APIがOcean OpticsをAndorとして報告する → 修正済み。**
   `_api_camera_backend()`/`_api_spectrometer_backend()`に`camera_oceanoptics`/
   `spectrometer_oceanoptics`モジュール判定分岐を追加し`"oceanoptics_seabreeze"`を返す。
   `_api_build_camera_info()`の`hardware_connected`は`thread.cam`だけでなく`thread.spec`も
   見るよう変更。`_api_build_spectrometer_info()`はbackendが`oceanoptics_seabreeze`の場合、
   常に`True`を返す`spec_ctrl.is_initialized`ではなく、カメラスレッドの実接続状態
   (`isRunning()`かつ`thread.spec is not None`)から`hardware_connected`を導出するよう変更。
   テスト: `tests/test_api_oceanoptics_mixin.py`の`BackendDetectionTests`/
   `HardwareConnectedDetectionTests`(7件)。

4. **P1: ウィザードの検出結果がOcean Opticsのserial欄へ反映されない → 修正済み。**
   `_PagePaths.apply_probe_result()`が`result["supplier"]`を見て、Ocean Opticsなら
   `_oo_serial`欄+`config["serial_number"]`、それ以外は従来通り`_pi_serial`欄+

   テスト: `tests/test_config_wizard_oceanoptics.py`(3件、単一/複数デバイス検出・

   `AcquisitionMixin._sync_fixed_spectrometer_center()`で、`physical_center_wl`と

### レビュー対応中に追加で発見した問題(範囲外だが同時に修正)

`.venv`で実際に`SpectrometerGUI(debug=True)`を`"model": "OceanOptics"`設定で構築し、
`init_finished`まで完走・1回取得・ウィジェット可視性を確認するスモークテストを実施した
結果(下記オープン課題1・2・3が事実上解消)、**軸ラベル/警告バナーがカメラ接続後に一度も
更新されない**という別のP1級不具合を発見した: `update_plot_labels()`は較正の適用/解除・
Raman/Wavelengthモード切替でしか呼ばれておらず、Ocean Opticsが接続完了しnative wavelength
軸が使えるようになった時点では一度も呼ばれない。そのため`lbl_axis_warning`も
プロット下軸ラベルも接続前の`"Pixel"`のまま固定されていた。
`AcquisitionMixin.on_camera_initialized()`の`_sync_fixed_spectrometer_center()`/
`_apply_hardware_capability_ui()`呼び出しの直後に`self.update_plot_labels()`を追加して
修正し、同スモークテストで`lbl_axis_warning visible: True`・
下軸ラベル`"Wavelength (nm) [Ocean Optics factory calibration]"`になることを確認した。

なお同じスモークテストで、`spectrometerConfig.json`に`"model": "OceanOptics"`でも
`"com_port": "COM3"`という無関係なキーが書き戻されることに気づいた。原因は
`ConfigMixin.load_spectrometer_config()`の`default_config`が全モデル共通で
`"com_port": "COM3"`を含み、読み込んだJSONにキーが無ければ無条件にマージされるためで、
**Ocean Optics固有ではなくAndor構成でも同様に発生する既存の挙動**(今回のレビュー対象外)。
実害はない(Ocean Optics側のコードは`com_port`を一切参照しない)ため、今回は修正せず記録
のみに留めた。
