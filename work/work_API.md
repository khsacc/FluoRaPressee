# FluoraPressée API化 実装計画

Generated by Claude Code

## 背景

FluoraPressée を同一LAN内の他PCから操作できるようにするため、FastAPI + uvicorn による
HTTP APIレイヤーを追加する。校正やROI設定など基本操作はアプリ内(GUI)で完結させ、ユーザーが
明示的に「APIサーバー起動」ボタンを押した場合のみAPIが立ち上がる想定。外部から可能にする操作は
以下の4種:
1. 校正パラメータの適用
2. データ取得(露光時間・積算数・バックグラウンド減算オプション付き)
3. データ取得+フィット(フィット範囲・関数を引数化)
4. データ取得+フィット+圧力算出(センサー・スケール・温度補正を引数化)

設計検討の過程で、既存コードに**カメラアクセスの排他制御が存在しない**ことが判明した
(`camera_andor.py`/`camera_princeton.py` の `acquire_single_image()` と Live View用の
`run()` ループが、ロック無しで同じ `self.cam.snap()` を呼びうる)。現状事故が起きていないのは
「校正ダイアログを開く前に必ずLive Viewを止める」という運用規約のみに支えられており、API化で
GUIとAPIの両方からトリガーされうる状況になるとこの前提が崩れる。そのため本計画は**まず
排他制御を修正してから**API本体に着手する。

また設計レビューの過程で、以下2点をユーザーと相談し方針を確定した(詳細は下記「採用する方針」参照):
- APIサーバー起動中は、GUIの測定トリガー・設定変更系の操作を丸ごとロックし、表示系の操作
  (プロットスタイル・自動レンジ)のみ有効なままにする。
- 背景(dark)減算は、露光時間等がミスマッチした場合デフォルトで拒否(エラー)する。加えて
  クライアントがdarkデータを直接送信できるモードを追加する。

なお、「リモートから新規にdarkを取得し直す」機能(API経由でのdark再取得)は本計画では
**実装しない**。励起光を物理的に遮断するシャッター等の制御が本アプリには一切実装されて
おらず、まずGUI側でのシャッター制御(または同等の遮光手順)の実装とテストが先に必要なため。
将来シャッター制御が実装された段階で、改めてAPI経由のdark再取得を検討する。

## 採用する方針

### レイヤー構成

```
[他PC] --HTTP--> FastAPI (uvicornスレッド, src/api/server.py)
                     |
                     | GuiBridge.call(fn)  (Qtシグナル経由でGUIスレッドへ処理を委譲)
                     v
        SpectrometerGUI (メインスレッド, src/ui.py + ui_mixins/*)
                     |
                     | 既存の thread-safe なメソッド (update_exposure等) / Lock
                     v
        CameraThreadAndor / CameraThreadPI (QThread, src/camera_andor.py 等)
```

### 排他制御(2レイヤー)

- **低レベル(`_hw_lock`)**: `camera_andor.py`/`camera_princeton.py` 内で、実際に
  `self.cam.snap()` や `_apply_camera_settings()` を呼ぶ箇所を `threading.Lock` で守る。
  「複数スレッドから同時にハードウェアへ触れない」ことだけを保証する。
- **高レベル(`_acquisition_gate`)**: `SpectrometerGUI` に持たせる `threading.Lock`。
  「GUIの手動測定・API経由の測定・校正ダイアログのいずれか1つだけが同時に測定権を持てる」
  ことを保証する。GUIの各トリガー地点(`check_bg_and_take_single` 等)とAPIの `api_acquire`
  の両方がこのゲートを通す。取得できなければ、GUI側はメッセージ表示、API側はHTTP 409を返す。
  これは「実際に競合しないことを保証する」ための最終防衛線であり、次項のUIロックとは役割が
  異なる(UIロックが無かった時代・APIサーバー起動前のローカル操作同士の排他や、複数のAPI
  リクエストが同時に届いた場合の排他にも効く)。

### API稼働中のGUI操作ロック

個々のAPIリクエスト実行中だけGUIをロックする案も検討したが、**APIサーバーが起動している間は
ずっとロックする**方式を採用する。理由: リクエストとリクエストの合間に操作者がグレーティング・
中心波長・ROI・校正・ロード済み背景などの「物理的/論理的な測定条件」をローカルから変更できて
しまうと、リモート側が最初に確認した前提(校正済みの状態)が本人の知らないところで崩れる。
排他ゲート(`_acquisition_gate`)だけでは「取得の瞬間の競合」は防げても、この種の「リクエストと
リクエストの間に起きる設定変更」は防げない。

`src/ui_mixins/sequential_mixin.py` の `set_ui_enabled_during_seq(enabled)` が、既に
「測定トリガー・分光器設定・BG設定・ROI・フィット設定を一括で無効化し、プロットの見た目系
(Line/Scatter, rescaleチェックボックス)だけ有効なまま残す」という、まさに欲しい制御を実装
済み(連続測定中に流用されている)なので、これをAPIサーバーの起動/停止にも流用する
(Step 9で実装)。ローカルの連続測定中ロックとAPI起動中ロックが両方同時にかかりうるため、
単純な bool ではなく「ロックをかけている理由の集合」で管理し、全ての理由が解消されたときのみ
再度有効化する設計にする。

### Darkデータの扱い

当初案(`dark.mode: "reuse_loaded"` で、ロード済み背景とのミスマッチがあっても警告フィールドを
付けて減算を続行する)は、暗電流が露光時間に比例しない成分(バイアス)を含むため、ミスマッチ時に
黙って誤った値を返すリスクがあると判断し、以下の3点に変更する:

1. `dark.mode: "reuse_loaded"` でロード済み背景とミスマッチ(露光時間・積算数・ROI等)した
   場合、**デフォルトで減算を実行せずHTTP 422エラーを返す**。どうしても近似的に続行したい
   場合のみ、リクエストに `dark.ignore_mismatch: true` を明示的に指定する(その場合も応答に
   警告フィールドを含める)。
2. `dark.mode: "provided"` を新設し、リクエストに `dark.data`(生の暗電流スペクトル配列)を
   直接含められるようにする。この場合ミスマッチチェックは行わず(クライアントが露光時間を
   合わせて自分で取得した前提)、配列長が一致するかだけを確認する。

「リモートから新規にdarkを取得し直す」機能(`POST /background/acquire` 相当)は、励起光を
物理的に遮断するシャッター制御が本アプリに未実装のため、本計画のスコープからは除外する
(上記「背景」参照)。

また、APIが返す `y`(背景減算後スペクトル)は、GUIの "Subtract background" ラジオボタン
(`radio_bg_on`)や "Flip X-axis" チェックボックス(`chk_flip_x`)の状態に**依存させない**。
従来案では `self.latest_1d_data`(GUI表示用に `update_display()` が計算した値)をそのまま
API応答に使う想定だったが、これだとAPI応答の中身が操作者のGUI操作によって意図せず変化して
しまう。そのためAPIは常に `self.raw_1d_data`(真の生データ)を起点に、リクエストの `dark`
パラメータだけに基づいて独自に減算を計算し、Flip Xも適用しない(x軸は較正済みなら較正後の
単位、そうでなければpixel番号のまま、常に昇順で返す)。前項のUIロックによりAPI稼働中は
`radio_bg_on`/`chk_flip_x` 自体が変更されなくなるため実害は小さいが、責務を明確に分離する
ために独立させておく。

### GuiBridge(スレッド橋渡し)とデッドロック回避

`SpectrometerGUI` の状態はほぼ全てQWidgetの値であり、Qt的にGUIスレッド以外から触るのは禁止
(クラッシュ・不定動作のリスク)。そこで、既存コードが `camera_thread` との通信に使っている
「pyqtSignalは別スレッドからemitしてもGUIスレッド側で安全にキュー処理される」という仕組みを、
API用にも `GuiBridge` として1つ追加する。

**重要な注意**: `GuiBridge.call(fn)` で GUIスレッドに処理を渡した先で「フレームが届くまで
ブロックして待つ」ような同期待ちを書いてはいけない。GUIスレッドのイベントループ自体が
止まってしまい、camera threadからの `data_ready` シグナル(GUIスレッドのイベントループが
処理するのを待っている)が永遠に処理されずデッドロックする。そのため取得系のAPI処理は
「GUIスレッド側は開始するだけして即座に戻る」「完了待ちは `concurrent.futures.Future` を使い
APIワーカースレッド側で行う」の2段階に分離する。完了通知は既存の `_process_completed_data`
(camera threadの `data_ready` シグナル経由で通常通りGUIスレッドが呼ぶ)に小さなフックを
追加し、そこから `Future.set_result()` する。

### GUIウィジェットを介さない計算

`DataAnalyzer.fit_spectrum`(`src/analysis.py`)と `PressureCalculator`
(`src/pressureCalc.py`)はQt非依存の純粋関数群なので、APIワーカースレッドから直接呼ぶ
(GuiBridge経由にしない)。これにより、他PCからのAPIリクエストが操作者のGUI上のフィット関数
選択や圧力計算ウィンドウの状態を勝手に書き換えることを避ける。校正の適用・現在状態の取得
(取得済み `calib_coeffs` 等の読み出し)はGUIの状態そのものを扱うためGuiBridge経由とする。

### 対話的ダイアログの回避

既存の `handle_bg_mismatch_and_run`(`src/ui_mixins/file_io_mixin.py`)はバックグラウンド
設定のミスマッチを検知すると `QMessageBox.exec()` でユーザーのクリックを待つ。この経路を
APIから使うと、操作者の目の前に理由不明なダイアログが出たままAPIリクエストがハングする。
そのため `api_acquire` はGUIのこの対話的経路を一切使わず、`check_bg_mismatch()` の真偽値
だけを確認し、ミスマッチ時はダイアログを出さず前項の方針(デフォルト422エラー)に従う。

### ステップ間の依存関係

Step 1〜9は基本的に**番号順に直列実行**する。特にStep 1・2(排他制御)はStep 5(`api_acquire`)
より前に完了している必要がある。Step 3(GuiBridge)とStep 4(積算数オーバーライド)は
互いに独立なので順序を入れ替えても良いが、番号順で進めるのが分かりやすい。Step 6以降は
Step 5までの内部APIが完成していることを前提とする。

### 各ステップ共通の検証手順

すべてのステップで、作業完了後に必ず以下を行う:

1. `python main.py --debug` を起動し、ウィンドウが例外なく開き "Camera Ready" になることを確認する。
2. そのステップに対応する検証項目(各Step末尾に記載)を実施する。
3. `git diff --stat` で意図したファイルだけが変更されていることを確認する。
4. Step 1・2については特に、既存のGUI操作フロー(Take single spectrum → 校正ダイアログを開く、
   Commence Measurement中に他の操作を試す等)が壊れていないことを手動で確認する。

## Step 1: カメラハードウェアアクセスの低レベル排他ロック

**依存:** なし(最初に実行する)

**対象ファイル:** `src/camera_andor.py`, `src/camera_princeton.py`

**実行プロンプト:**

> `src/camera_andor.py` の `CameraThreadAndor` と `src/camera_princeton.py` の
> `CameraThreadPI` に、実機ハードウェアへの排他アクセスを保証するロックを追加する。
>
> 背景: 両クラスとも `run()` メソッドの `is_measuring` 分岐(Live View用の連続取得ループ)と
> `acquire_single_image()`(校正ダイアログ等が使う同期単発取得メソッド)の両方が、それぞれ
> 独立に `self._apply_camera_settings()` と `self.cam.snap()` を呼んでいるが、この2つの
> 呼び出しは現在どちらもロックで保護されていない。両者が異なるスレッドから同時に呼ばれると
> (例: Live View実行中に別スレッドから `acquire_single_image()` が呼ばれると)、同じ
> ハードウェアハンドルに対して競合したアクセスが発生しうる。既存の `self._lock`
> (`threading.Lock`)はフラグの読み書きだけを短時間保護する用途で使われており、
> `cam.snap()` のような長時間(露光時間+タイムアウト分)ブロックする呼び出しをこの
> `self._lock` で保護すると、`update_exposure()` 等の他の操作がその間ずっと待たされてしまう。
>
> そこで両クラスの `__init__` に新規 `self._hw_lock = Lock()`(`threading` の `Lock`、
> 既存の `self._lock` とは別インスタンス)を追加し、以下の箇所を `with self._hw_lock:` で
> 囲む:
> - `run()` 内、`is_measuring` が True の分岐での `self._apply_camera_settings()` 呼び出し
>   (デバッグモードでない場合)と、その後の `self.cam.snap(...)` 呼び出し
> - `acquire_single_image()` 内の `self._apply_camera_settings()` 呼び出しと
>   `self.cam.snap(...)` 呼び出し
>
> `time.sleep(...)` の呼び出しは `_hw_lock` の**外側**に置くこと(ロックを持ったままsleepしない)。
> デバッグモード(`self.debug=True`)のダミーデータ生成部分はハードウェアに触らないので
> ロック不要。
>
> 作業後、`python main.py --debug` でアプリを起動し、"Take single spectrum" と
> "Commence Measurement" が問題なく動作すること、"Calibrate x-axis" ボタンから校正ダイアログを
> 開いて閉じる一連の操作(内部で `acquire_single_image()` を繰り返し呼ぶ)が今まで通り
> 動作することを確認する。可能であれば実機(Andor or Princeton、`--debug` を付けずに起動)でも
> 同様の確認を行う。

## Step 2: GUI操作レベルの排他ゲート導入

**依存:** Step 1

**対象ファイル:** `src/ui.py`, `src/ui_mixins/acquisition_mixin.py`,
`src/ui_mixins/file_io_mixin.py`, `src/ui_mixins/spectrometer_control_mixin.py`

**実行プロンプト:**

> Step 1で追加したハードウェアレベルのロックとは別に、「GUIの手動測定・校正ダイアログ・
> (将来追加する)API経由の測定のうち、同時にどれか1つだけが測定権を持てる」ことを保証する
> 論理的な排他ゲートを `SpectrometerGUI` に追加する。
>
> 1. `src/ui.py` の先頭に `import threading` を追加(未import であれば)。`__init__` の
>    状態初期化部分(他の `self.xxx = None` 等が並んでいる箇所)に
>    `self._acquisition_gate = threading.Lock()` を追加する。
>
> 2. `src/ui_mixins/acquisition_mixin.py` の `AcquisitionMixin` に、ゲートの取得・解放を行う
>    小さなヘルパーメソッドを2つ追加する:
>    - `_try_acquire_gate(self) -> bool`: `self._acquisition_gate.acquire(blocking=False)` を
>      試み、成功したら `self._gate_held_by_me = True` をセットして `True` を返す。失敗したら
>      `False` を返す(既に誰かが測定中)。
>    - `_release_acquisition_gate(self) -> None`: `getattr(self, '_gate_held_by_me', False)` が
>      True の場合のみ `self._gate_held_by_me = False` としてから `self._acquisition_gate.release()`
>      する(二重解放によるRuntimeErrorを防ぐため、保持フラグで必ずガードすること)。
>
> 3. ゲートの取得(失敗時は早期return + ユーザーへの通知)を以下の箇所に追加する:
>    - `src/ui_mixins/file_io_mixin.py` の `check_bg_and_take_single`, `check_bg_and_start_meas`,
>      `check_bg_and_start_seq`: それぞれ内部で `take_single_spectrum`/`start_measurement`/
>      `start_sequential` を呼ぶ前に `self._try_acquire_gate()` を試み、失敗したら
>      `QMessageBox.warning(self, "Busy", "Another acquisition is already in progress.")` を
>      出して return する(既存の `handle_bg_mismatch_and_run` によるコールバック呼び出しの
>      「前」でゲートを取得すること)。
>    - `src/ui_mixins/file_io_mixin.py` の `on_acq_bg_clicked`: 同様にゲート取得を試み、
>      失敗したら同じ警告を出して return する。
>    - `src/ui_mixins/spectrometer_control_mixin.py` の `on_calibrate_neon`: 校正ダイアログを
>      開く前にゲート取得を試み、失敗したら警告を出して return する(ダイアログを開かない)。
>      ダイアログが閉じた後(`calib_win.exec()` の直後)にゲートを解放する。
>
> 4. ゲートの解放を以下の箇所に追加する:
>    - `src/ui_mixins/acquisition_mixin.py` の `stop_measurement`: 末尾でゲートを解放する
>      (連続測定・単発測定のいずれの停止でも呼ばれる共通の停止経路)。
>    - `src/ui_mixins/acquisition_mixin.py` の `_process_completed_data`: `is_single_shot` の
>      分岐内、単発測定が完了した直後にゲートを解放する(このメソッドは
>      `stop_measurement()` を経由しない単発取得の完了パスであるため、これも別途必要)。
>
> 5. 既知の限界としてコード中にコメントで残す: Andorカメラ(`camera_andor.py`)は連続エラーで
>    自動停止した際(`_consec_errors >= 5`)に `is_measuring` を内部で False にするだけで
>    GUI側に通知するシグナルが無い(Princeton版の `acquisition_failed` シグナルに相当するものが
>    Andor版に無い、既存の非対称性)。この場合ゲートが解放されないまま残留する可能性がある。
>    本ステップではこの根本修正は行わず、`_release_acquisition_gate` 呼び出し箇所の近くに
>    1行コメントで既知の制限として明記するだけに留める。
>
> 6. なお、本Stepで追加する `_acquisition_gate` は、Step 9で追加する「APIサーバー起動中は
>    GUIの測定系コントロールを丸ごと無効化する」という広いUIロックとは別レイヤーである。
>    前者は「実際に競合しないことを保証する」ための最終防衛線(ウィジェットの有効/無効に
>    関わらず効く)、後者は「操作者に触らせないようにする」ためのUI制御であり、役割が異なる
>    ため両方必要になる。本Stepでは広いUIロックの実装は行わない(Step 9で行う)。
>
> 作業後、`python main.py --debug` で以下を確認する: (a) "Take single spectrum" 実行中に
> もう一度同じボタンを連打してもクラッシュしないこと、(b) "Commence Measurement" 中に
> "Calibrate x-axis" を押すと「Busy」警告が出て校正ダイアログが開かないこと、(c) 測定を
> "Terminate Measurement" で止めた後は再度 "Take single spectrum" や校正ダイアログが正常に
> 開けること(ゲートが正しく解放されていること)。

## Step 3: スレッド橋渡し基盤 GuiBridge の追加

**依存:** なし(Step 1・2と並行しても良いが、番号順推奨)

**対象ファイル(新規):** `src/api/__init__.py`, `src/api/gui_bridge.py`
**対象ファイル(変更):** `main.py`

**実行プロンプト:**

> APIワーカースレッドからPyQtのGUIスレッドへ処理を安全に橋渡しするための `GuiBridge` を追加する。

>
> 2. 新規 `src/api/gui_bridge.py` に `GuiBridge(QObject)` クラスを実装する:
>    - `pyqtSignal(object, object)` を1つ持つ(呼び出したい関数と、結果を受け取るための

>    - コンストラクタでそのシグナルを自分自身のスロットに接続する(別スレッドからemitされる前提の
>      ため、Qtのデフォルト接続方式で自動的にQueuedConnectionとなり、GUIスレッドのイベント
>      ループ上でスロットが実行される)。

>      発生した場合は `future.set_exception(...)` にセットする。
>    - 呼び出し用の公開メソッド `call(self, fn, timeout=60)` を用意する: 内部で
>      `concurrent.futures.Future()` を作り、シグナルをemitし、`future.result(timeout=timeout)`
>      を呼んで結果を返す(GUIスレッド以外から呼ばれることを想定。GUIスレッド自身から呼ぶと
>      自分自身の完了を待つことになりデッドロックするため、`call()` の先頭で

>    `src/api/server.py` から `window.gui_bridge` としてアクセスする)。

## Step 5: api_acquire の実装(Futureベースの非同期取得ブリッジ、dark処理を含む)

**依存:** Step 1, 2, 3, 4

**対象ファイル(新規):** `src/ui_mixins/api_mixin.py`
**対象ファイル(変更):** `src/ui.py`, `src/ui_mixins/acquisition_mixin.py`

**実行プロンプト:**

> APIから呼び出せる同期的な単発取得メソッド `api_acquire` を実装する。これは
> Step 3のドキュメントに記載した「GUIスレッド側は開始するだけで即座に戻り、完了待ちは
> APIワーカースレッド側で `Future` を使う」という2段階設計に従う必要がある。GUIスレッド上で
> フレーム到着までブロックして待つコードを書くと、`data_ready` シグナル(GUIスレッドの
> イベントループが処理する)が永久に処理されずデッドロックするため、絶対に避けること。
>
> **darkの扱いについての重要な設計方針**(詳細は `work/work_API.md` の「採用する方針」内
> 「Darkデータの扱い」を参照): 背景減算はGUIの `radio_bg_on`/`chk_flip_x` に一切依存させず、
> `api_acquire` がリクエストの `dark_mode` パラメータだけに基づいて独自に計算する。また
> `dark_mode="reuse_loaded"` でミスマッチが起きた場合はデフォルトで例外を送出し(黙って
> 誤った減算をしない)、`ignore_mismatch=True` が明示された場合のみ警告付きで継続する。
>
> 1. 新規 `src/ui_mixins/api_mixin.py` に `class ApiMixin:` を作成する。ファイル冒頭に
>    `class BackgroundMismatchError(Exception): pass` を定義する(`dark_mode="reuse_loaded"`
>    でミスマッチかつ `ignore_mismatch=False` の場合に送出する専用例外。`src/api/server.py`
>    側でこれを捕捉してHTTP 422に変換できるようにするため)。
>
> 2. `ApiMixin` に以下の2メソッドを実装する:
>    - `_api_start_acquire(self, exposure_s=None, accumulations=None) -> concurrent.futures.Future`:
>      **GUIスレッド上で呼ばれる**(=`GuiBridge.call()` から呼ばれる)ことを前提とした
>      メソッド。dark処理は一切行わない(生データの取得だけを担当する)。処理:
>      a. `self._try_acquire_gate()`(Step 2で追加)を試み、失敗したら
>         `RuntimeError("acquisition busy")` を送出する。
>      b. `exposure_s` が指定されていれば `self.thread.update_exposure(exposure_s)` を呼び、
>         既存の `acquire_single_image()`(`camera_andor.py`)が行っているのと同様に
>         短い `time.sleep(0.1)` で設定反映を待つ。
>      c. `accumulations` が指定されていれば `self._active_target_accum = accumulations`
>         (Step 4)。
>      d. `concurrent.futures.Future()` を1つ作り、`self._api_pending_future = future` に
>         保存する。
>      e. `self.take_single_spectrum()`(既存の `AcquisitionMixin` のメソッド、ゲート取得は
>         行わない実装のままなのでそのまま呼んでよい、二重にゲートを取ろうとしていないか
>         確認すること)を呼ぶ。
>      f. `future` を return する(ここではまだ結果は入っていない)。
>    - `api_acquire(self, exposure_s=None, accumulations=None, dark_mode="none", dark_data=None, ignore_mismatch=False, timeout=30.0) -> dict`:
>      **APIワーカースレッド上で呼ばれる**ことを前提としたメソッド。処理:
>      a. `dark_mode == "provided"` かつ `dark_data is None` の場合は `ValueError` を送出する。
>      b. `future = self.gui_bridge.call(lambda: self._api_start_acquire(exposure_s, accumulations))`
>         で開始処理をGUIスレッドに委譲する。`RuntimeError("acquisition busy")` が送出された
>         場合はそのままAPIハンドラ側に伝播させる(呼び出し元でHTTP 409に変換)。
>      c. `result = future.result(timeout=timeout)` でGUIスレッドからの完了通知(次項)を待つ
>         (このブロッキングはAPIワーカースレッド上で起きるのでGUIは固まらない)。
>         `result` は `{"raw": <numpy配列>, "mode": "1d"|"2d"}`。
>      d. `mode == "1d"` の場合のみ、`raw` に対して `dark_mode` に応じた処理を行う:
>         - `"none"`: `y = raw.copy()`。
>         - `"provided"`: `dark_arr = np.asarray(dark_data)`、長さが `raw` と一致しなければ
>           `ValueError`。一致すれば `y = raw - dark_arr`。
>         - `"reuse_loaded"`: `self.loaded_bg_data` が `None` なら `ValueError`。
>           `self.check_bg_mismatch()`(この時点でAPI稼働中はStep 9のUIロックにより
>           `loaded_bg_data` 等が変更されないことが保証されているため、GuiBridge経由に
>           せず直接呼んで良い)を確認し、ミスマッチかつ `ignore_mismatch is False` の場合は
>           `BackgroundMismatchError` を送出する。ミスマッチかつ `ignore_mismatch is True` の
>           場合は減算した上で戻り値に `"background_mismatch_warning": True` を含める。
>           ミスマッチが無ければ通常通り `y = raw - self.loaded_bg_data` するだけで警告は
>           付けない。
>         `mode == "2d"` の場合はdark減算をスキップし、`y = raw` のまま返す。
>      e. `x = self.gui_bridge.call(lambda: self.get_x_axis(len(raw)))`(較正係数があれば
>         較正後の単位、無ければpixel番号。Flip Xは適用しない。`get_x_axis` は
>         `self.radio_spec_mode_raman` 等のウィジェットを読むためGuiBridge経由にすること)。
>      f. 最終的な dict(`x`, `y_raw`(=raw), `y`, `mode`, `exposure_time_s`, `accumulations`,
>         `detector_temperature_c`, `timestamp`, 該当すれば `background_mismatch_warning`)を
>         組み立てて return する。
>
> 3. `src/ui_mixins/acquisition_mixin.py` の `_process_completed_data` の単発測定完了パスに、
>    Step 2・4で追加した処理に続けて以下を追加する(dark処理は含めない、生データを渡すだけ):
>    ```
>    pending = getattr(self, '_api_pending_future', None)
>    if pending is not None:
>        self._api_pending_future = None
>        if not pending.done():
>            pending.set_result({
>                "raw": (self.raw_1d_data if mode == "1d" else self.raw_2d_data),
>                "mode": mode,
>            })
>    ```
>
> 4. `src/ui.py` の `class SpectrometerGUI(...)` の継承リストに `ApiMixin` を追加し、
>    `from src.ui_mixins.api_mixin import ApiMixin` をimportする。`__init__` に
>    `self._api_pending_future = None` の初期化を追加する。
>
> 5. 動作確認: `main.py` に一時的な検証コードを追加する。`threading.Thread` を起動し、
>    `window.gui_bridge = bridge`(Step 3で設定済み)を使って:
>    (a) `window.api_acquire(exposure_s=0.2, accumulations=2, dark_mode="none")` が例外なく
>    生データを返すこと、(b) 呼び出し中もGUIウィンドウがフリーズしないこと、(c) GUIから
>    "Acquire and save background" で背景を保存・ロードした状態で
>    `dark_mode="reuse_loaded"` を呼ぶと正しく減算されること、(d) 露光時間を変えて
>    `exposure_s` を渡す(ミスマッチを起こす)と `BackgroundMismatchError` が送出されること、
>    (e) `ignore_mismatch=True` を渡すと警告フラグ付きで減算が続行されること、を確認する。
>    確認後、この検証用コードは削除する。

## Step 6: api_fit / api_pressure / api_apply_calibration / api_get_status の実装

**依存:** Step 5

**対象ファイル:** `src/ui_mixins/api_mixin.py`

**実行プロンプト:**

> `ApiMixin` に、フィッティング・圧力計算・校正適用・状態取得の4メソッドを追加する。
>
> 1. `api_fit(self, x, y, fit_function, fit_start=None, fit_end=None) -> dict`:
>    GUIスレッドに委譲する必要はない(`DataAnalyzer` はQt非依存)。`self.analyzer.fit_spectrum(
>    np.asarray(x), np.asarray(y), fit_function, fit_start, fit_end)` をそのまま呼び、
>    戻り値 `(x_fit, y_fit_curve, res)` を dict にまとめて返す(`res` が `None` ならフィット
>    失敗を示すエラー情報を返す)。numpy配列はリストに変換しなくてよい(呼び出し元のAPI層で
>    JSON化する)。
>
> 2. `api_pressure(self, peak, peak_err, sensor, pressure_scale, zero_pressure_peak, temperature_correction=None) -> dict`:
>    同様にGUIスレッド不要。`PressureCalculator` の3メソッド(`is_temp_in_range`/
>    `get_corrected_lam0`/`calculate`)はすべてキーワード専用引数(`*`マーカー)なので、呼び出しは
>    必ず `PressureCalculator.calculate(sensor=..., p_scale=..., lam=..., lam0=..., lam0_at_t0=...,
>    lam_err=..., current_t=..., t0=...)` のようにキーワードで書くこと(位置引数で呼ぶと
>    `TypeError` になる)。`lam0_at_t0` は温度補正が有効かどうかに関わらず、`temperature_correction`
>    が渡されていれば常に `temperature_correction["zero_pressure_peak_at_t0"]` を使う(渡されて
>    いなければ省略してよく、その場合 `calculate()` 内部で `lam0` と同値として扱われる)。
>    `temperature_correction.get("enabled")` が真の場合のみ `PressureCalculator.is_temp_in_range(...)`
>    (有効範囲外でも計算は続行し警告のみ付与)→ `PressureCalculator.get_corrected_lam0(...)` で
>    `lam0` を補正してから `PressureCalculator.calculate(...)` を呼ぶ。`src/pressureCalc_ui.py` の
>    `PressureCalculatorWindow.calculate()` と全く同じ計算順序に倣うこと(このメソッドを直接参照
>    して処理順序を確認すること)。
>    戻り値は `pressure`, `pressure_err`, `temperature_warning`(範囲外の場合の文字列 or None)を
>    含むdict。
>
> 3. `api_apply_calibration(self, c0, c1, c2, unit, laser_wavelength_nm=None, label="api") -> dict`:
>    GUIの表示ラベル(`self.lbl_loaded_calib`)を更新するため、GuiBridge経由で呼ぶ必要がある
>    (このメソッド自体は「GUIスレッド上で呼ばれる」前提で書き、`src/api/server.py` 側で
>    `gui_bridge.call(lambda: window.api_apply_calibration(...))` として呼ぶ)。内部では
>    `src/ui_mixins/file_io_mixin.py` の `apply_calibration(self, coeffs, filename, calib_unit, calib_laser_wl)`
>    をそのまま呼ぶだけでよい(`coeffs=(c0, c1, c2)`, `filename=label`)。
>
> 4. `api_get_status(self) -> dict`: GUIスレッド上で呼ばれる前提。カメラ接続状態
>    (`hasattr(self, 'thread')` かつ `self.thread.isRunning()`)、現在の露光時間
>    (`self.spin_acq_time.value()`)、校正の有無(`self.calib_coeffs is not None`)と
>    unit・label、ROI設定、ロード済み背景の有無とメタデータ、
>    `self._acquisition_gate.locked()` を busy 状態として、dict化して返す。
>
> 動作確認は Step 5と同様、一時的な検証コードで `api_fit`/`api_pressure`/`api_get_status` を
> 直接呼び出し(GUIスレッド不要なものはメインスレッドから直接、`api_apply_calibration` は
> `gui_bridge.call()` 経由で)、期待通りの値が返ることを確認してから検証コードを削除する。

## Step 7: FastAPIアプリ本体・スキーマ・認証

**依存:** Step 5, 6

**対象ファイル(新規):** `src/api/schemas.py`, `src/api/server.py`, `requirements.txt`

**実行プロンプト:**

> `ApiMixin` の各メソッドをHTTPエンドポイントとして公開するFastAPIアプリを実装する。
>
> 1. 新規 `requirements.txt` を作成し、既存の実行時依存(`PyQt5`, `pyqtgraph`, `numpy`, `scipy`,
>    `pylablib`, `pyserial` — READMEに記載されているもの)に加えて `fastapi`, `uvicorn`,
>    `pydantic` を追加する。
>
> 2. 新規 `src/api/schemas.py` に、以下のPydanticモデルを定義する:
>    - `CalibrationRequest`: `c0: float, c1: float, c2: float, unit: Literal["Wavelength","Raman shift"], laser_wavelength_nm: float | None = None, label: str = "api"`
>      (unit が "Raman shift" のとき `laser_wavelength_nm` が必須になるバリデータを追加)
>    - `DarkOptions`: `mode: Literal["none", "reuse_loaded", "provided"] = "none", data: list[float] | None = None, ignore_mismatch: bool = False`
>      (`mode == "provided"` のとき `data` が必須になるバリデータを追加)
>    - `AcquireRequest`: `exposure_time_s: float | None = None, accumulations: int = 1, dark: DarkOptions = DarkOptions()`
>    - `FitRange`: `start: float, end: float`
>    - `AcquireFitRequest(AcquireRequest)`: `fit_function: Literal["Gauss","Lorentz","Pseudo Voigt","Double Gauss","Double Lorentz","Double pseudo Voigt"], fit_range: FitRange | None = None`
>    - `TemperatureCorrection`: `enabled: bool, scale: str, current_t: float, t0: float, zero_pressure_peak_at_t0: float`
>    - `AcquirePressureRequest(AcquireFitRequest)`: `sensor: str, pressure_scale: str, zero_pressure_peak: float, temperature_correction: TemperatureCorrection | None = None`
>    - 対応するレスポンス用モデルも定義する(前回の会話で確定したJSON形状に合わせる:
>      `/acquire` は x/y_raw/y/exposure_time_s/accumulations/detector_temperature_c/timestamp/
>      background_mismatch_warning(任意)、`/acquire/fit` はそれに fit を追加、
>      `/acquire/pressure` はさらに pressure_gpa/pressure_err_gpa/temperature_warning を追加)。
>
> 3. 新規 `src/api/server.py` に `create_app(gui_window, gui_bridge, api_key) -> FastAPI` という
>    ファクトリ関数を実装する:
>    - `X-API-Key` ヘッダーを検証する `Depends` 関数を定義し、全ルートに適用する(不一致なら
>      HTTP 401)。
>    - `GET /status`: `gui_bridge.call(gui_window.api_get_status)` を呼んでそのまま返す。
>    - `POST /calibration`: `gui_bridge.call(lambda: gui_window.api_apply_calibration(...))`。
>    - `POST /acquire`: `gui_window.api_acquire(...)` を直接呼ぶ(このメソッド自体が内部で
>      `gui_bridge` を使うのでAPIハンドラ自身はGuiBridge経由にしない、Step 5の設計通り)。
>      `RuntimeError("acquisition busy")` → HTTP 409。`ApiMixin` の
>      `BackgroundMismatchError` → HTTP 422(エラーメッセージにミスマッチの詳細を含める)。
>      `ValueError`(`dark_data` 未指定・長さ不一致など)→ HTTP 400。
>      `concurrent.futures.TimeoutError` → HTTP 504。
>    - `POST /acquire/fit`: `/acquire` と同じ取得・dark処理の後、`gui_window.api_fit(...)` を
>      呼んで結果を合成する。
>    - `POST /acquire/pressure`: 上記に加え `gui_window.api_pressure(...)` を呼ぶ。
>    - 各エンドポイントの numpy 配列は `.tolist()` でJSONシリアライズ可能な形にしてから返す。
>
> 4. まだGUIへのStart/Stopボタンは実装しない(次のStep 8)。代わりに、動作確認用として
>    一時的なスクリプト `scripts/_dev_run_api.py`(gitで管理してよい一時スクリプトとして
>    残すか、確認後に削除するかはユーザーに確認)を作成し、`main.py` 相当の初期化
>    (QApplication, GuiBridge, SpectrometerGUI, `--debug`固定)を行った上で、別
>    `threading.Thread` で `uvicorn.run(create_app(window, bridge, api_key="devkey"), host="127.0.0.1", port=8765)`
>    を起動するようにする。
>
> 5. 動作確認: 上記の一時スクリプトを実行し、別ターミナルから `curl` で
>    `GET /status`, `POST /calibration`, `POST /acquire`, `POST /acquire/fit`,
>    `POST /acquire/pressure` を一通り叩き、期待通りのJSONが返ること、GUIウィンドウがその間
>    フリーズしないことを確認する。認証ヘッダーを外すと401になること、
>    `dark.mode="reuse_loaded"` でわざとミスマッチさせると422になることも確認する。

## Step 8: GUI統合 — Start/Stop APIサーバーボタンと稼働中のUIロック

**依存:** Step 7

**実行プロンプト:**

> Step 7で作った `create_app` を、GUIから明示的に起動・停止できるようにする。あわせて、
> 「採用する方針」の「API稼働中のGUI操作ロック」に記載した通り、APIサーバーが起動している間は
> GUIの測定系コントロールを丸ごと無効化し、表示系(プロットスタイル・自動レンジ)だけ操作可能
> な状態にする。
>
> 1. `src/ui_mixins/sequential_mixin.py` の `set_ui_enabled_during_seq` を、複数の理由が
>    同時にロックをかけられるように改修する。具体的には `SpectrometerGUI.__init__` に
>    `self._ui_lock_reasons = set()` を追加し、ロックをかけたい箇所は
>    `self._ui_lock_reasons.add("sequential")` のように理由を追加してから
>    `set_ui_enabled_during_seq(False)` を呼び、解除時は `self._ui_lock_reasons.discard("sequential")`
>    してから、`len(self._ui_lock_reasons) == 0` の場合のみ `set_ui_enabled_during_seq(True)` を
>    呼ぶようにする(既存の `start_sequential`/`stop_sequential` 内の呼び出し箇所を、直接
>    `set_ui_enabled_during_seq(...)` を呼ぶ代わりにこの理由集合を経由する小さなヘルパー
>    `_lock_ui(reason)`/`_unlock_ui(reason)` 経由に置き換える)。
>
> 2. `src/ui_mixins/api_mixin.py` の `ApiMixin` に以下を追加する:
>    - `start_api_server(self, host, port)`: APIキーを `secrets.token_urlsafe(24)` 等で
>      ランダム生成して `self._api_key` に保持し、`create_app(self, self.gui_bridge, self._api_key)`
>      で作った FastAPI app を渡して `uvicorn.Server(uvicorn.Config(app, host=host, port=port,
>      log_level="info"))` を作成、これを `threading.Thread(target=server.run, daemon=True)` で
>      起動して `self._api_server = server`, `self._api_server_thread = thread` に保持する。
>      続けて `self._lock_ui("api_server")` を呼ぶ。
>    - `stop_api_server(self)`: `self._api_server.should_exit = True` をセットしてから
>      `self._api_server_thread.join(timeout=5)` する。続けて `self._unlock_ui("api_server")` を
>      呼ぶ。
>
> 3. `src/ui.py` の `__init__` に、コントロールパネルの末尾あたり(圧力計算ボタンの下等)に
>    "API Server" というQGroupBoxを追加する。中身: ポート番号入力欄(`CustomSpinBox`、
>    デフォルト8765)、"Start API Server" ボタン、"Stop API Server" ボタン(初期状態
>    disabled)、現在の状態・URL・APIキーを表示する複数行の `QLabel`(初期は "Not running")。
>    このグループボックス自体は `_lock_ui`/`_unlock_ui` の対象に含めない(Stopボタンは常に
>    押せる必要があるため)。
>
> 4. Start ボタン押下時: `start_api_server()` を呼び、成功したら状態ラベルに
>    `http://<ローカルIPアドレス>:<port>/docs` の形式のURLと生成されたAPIキーを表示する
>    (ローカルIPアドレスの取得は `socket.gethostbyname(socket.gethostname())` 等の簡易な方法で
>    良い、取得失敗時は `127.0.0.1` にフォールバック)。Stop ボタンを enabled にし、Start
>    ボタンとポート入力欄を disabled にする。
>
> 5. Stop ボタン押下時: `stop_api_server()` を呼び、状態ラベルを "Not running" に戻し、
>    ボタンの有効/無効を元に戻す。
>
> 6. `src/ui.py` の `closeEvent` で、`getattr(self, '_api_server', None)` が起動中であれば
>    ウィンドウを閉じる前に `stop_api_server()` を呼ぶ処理を追加する。
>
> 7. `main.py` に `window.gui_bridge = bridge` の配線(Step 3で追加済みのはず、無ければここで
>    追加)を確認する。Step 7で作った一時スクリプト `scripts/_dev_run_api.py` は、この時点で
>    役目を終えるため削除してよいか作業ログに記載する(削除するかどうかは実装者の判断でよい)。
>
> 動作確認: `python main.py --debug` で起動し、GUIの "Start API Server" ボタンを押して
> URLとAPIキーが表示されることを確認する。この状態で (a) "Take single spectrum" 等の測定系
> ボタンが無効化されていること、(b) プロットスタイルのLine/Scatter切り替えや自動レンジの
> チェックボックスは操作可能なままであること、(c) 別ターミナルから `curl` で `GET /status` 等
> を叩いて応答が返ること、(d) "Stop API Server" を押すと測定系ボタンが再度有効になり、その後
> の `curl` は接続エラーになることを確認する。さらに、ローカルで "Commence Measurement"
> (連続測定)を開始した状態でAPIサーバーを起動しようとした場合の挙動(両方の理由でロックが
> 掛かり、連続測定を止めるまでAPIからの測定はStep 2のゲートにより待たされる、または明示的に
> 拒否される)についても実機/デバッグモードで意図通りか確認する。
