# Adding a page to docs-site/

`docs-site/` is a Docusaurus site (single locale `ja`, see `docusaurus.config.ts`) that publishes
FluoRaPressée's online manual. The sidebar is **fully autogenerated from the filesystem**
(`sidebars.ts`: `tutorialSidebar: [{type: 'autogenerated', dirName: '.'}]`) — you never hand-edit
`sidebars.ts` to add a page. Ordering and grouping instead come from each file's frontmatter and each
category folder's `_category_.json`. Getting those two right is the entire task.

## 1. Decide where the page belongs

- **New page in an existing category** (`docs/api/`, `docs/data-formats/`) — e.g. one more endpoint,
  one more file format. Most requests are this. Go to [§2](#2-add-a-page-inside-an-existing-category).
- **New top-level page** (sibling of `intro.md`, `installation.md`, `hardware-config.md`,
  `basic-usage.md`) — a manual section that isn't "one more item in an existing list". Go to
  [§3](#3-add-a-new-top-level-page).
- **New category** (a whole new grouped section, like `api/` or `data-formats/` were when first
  created) — only when the content doesn't fit any existing category and is itself multi-page. Go to
  [§4](#4-add-a-new-category).

## 2. Add a page inside an existing category

1. Create `docs-site/docs/<category>/<kebab-case-name>.md`.
2. Frontmatter — `sidebar_position` is scoped **per folder**, not global. Check current values first:
   ```bash
   grep -m1 "^sidebar_position:" docs-site/docs/<category>/*.md
   ```
   Current state (re-check before relying on this — it drifts):
   - `docs/api/`: index=1, status=2, hardware=3, config=4, configurations=5, calibration=6, acquire=7,
     acquire-fit=8, acquire-pressure=9, errors=10, examples=11.
   - `docs/data-formats/`: index=1, spectrum-data=2, background=3, fitting-results=4,
     sequential-log=5, configuration=6.
   Pick the next free integer, or renumber neighbors if the new page belongs in the middle
   (Docusaurus only cares about relative order, not contiguity, but keep it tidy).
3. Required frontmatter fields: `sidebar_position`, `title`, `description`. Don't add `slug` (only
   `intro.md` uses it, to make itself the site root `/`).
   ```markdown
   ---
   sidebar_position: 12
   title: "POST /whatever"
   description: 一言で何のエンドポイントか
   ---

   # POST /whatever
   ```
   Quote `title` if it contains a `/` or special YAML characters (see `api/acquire.md`,
   `api/status.md`).
4. If this is an API endpoint page, match the existing per-endpoint page structure (`api/acquire.md`
   etc.: リクエスト example JSON, response, error cases) and also update `api/index.md`'s endpoint
   summary and `api/examples.md`/`api/errors.md` if the new endpoint needs a curl example or
   introduces a new error code. Also check whether `work/work_API.md` (implementation history/design
   rationale, per root `CLAUDE.md`) should record why the endpoint was added — that file lives outside
   `docs-site/` and is a separate, non-Docusaurus doc.

## 3. Add a new top-level page

Same as §2 but `sidebar_position` is shared across **all** top-level items — both loose `.md` files
*and* category folders (whose position comes from `_category_.json`, not frontmatter). Check both:
```bash
grep -m1 "^sidebar_position:" docs-site/docs/*.md
grep -m1 '"position"' docs-site/docs/*/_category_.json
```
Current state: intro=1, installation=2, hardware-config=3, basic-usage=4, `api/` category=5,
`data-formats/` category=6. A new top-level page/category takes the next integer, or you renumber to
insert it earlier.

## 4. Add a new category

Mirror `docs/data-formats/` (the simplest existing example):
1. `docs-site/docs/<new-category>/_category_.json`:
   ```json
   {
     "label": "表示ラベル",
     "position": 7,
     "link": {
       "type": "doc",
       "id": "<new-category>/index"
     }
   }
   ```
   `position` competes in the same top-level ordering as §3 — check both loose pages and existing
   category positions first.
2. `docs-site/docs/<new-category>/index.md` with `sidebar_position: 1`, acting as the category's
   landing page (compare `data-formats/index.md`: a short intro + a table linking every page in the
   category).
3. Add the category's other pages per §2, `sidebar_position` 2, 3, ... within the new folder.

## Content conventions

- **Language**: page content is Japanese (site `i18n.defaultLocale` is `ja`, single-locale — there is
  no English variant to keep in sync). Match the terse, direct style of existing pages.
- **Admonitions**: `:::note`, `:::caution` are already in use (see `hardware-config.md`, `intro.md`);
  `:::tip`/`:::info`/`:::danger` are standard Docusaurus admonitions and safe to use if a page needs
  them.
- **Cross-links**: use relative `.md` paths, resolved from the linking file's own directory — e.g.
  `[Configurationファイル](data-formats/configuration.md)` from `hardware-config.md` (same dir level),
  `[API](../api/index.md)` from `data-formats/index.md` (one level up then into a sibling category).
  **`docusaurus.config.ts` sets `onBrokenLinks: 'throw'`** — a broken relative link fails `npm run
  build` outright, which is also exactly what `test-docs.yml` runs on every PR touching `docs-site/**`.
  Double-check the relative path resolves before considering the page done; don't guess.
- **Images**: put new screenshots under `docs-site/static/img/manual/` (existing manual screenshots
  live there). Reference them from a docs page with a site-root-relative path, e.g.
  `![説明](/img/manual/YourScreenshot.png)` — Docusaurus auto-prepends `baseUrl`
  (`/FluoRaPressee/`) to paths starting with `/` in Markdown, so don't hardcode the baseUrl yourself.
  This differs from how the root `README.md`/`README_ja.md` embed the same images
  (`docs-site/static/img/manual/....jpg`, a plain repo-relative path that only works because GitHub
  renders those files directly, not through Docusaurus) — don't copy that path style into a page under
  `docs-site/docs/`.
- **Linking back from the root README**: per root `CLAUDE.md`, `README_ja.md` is the actively
  maintained doc (`README.md`'s feature list is HTML-commented out and unrendered — don't bother
  mirroring there). If the new page is worth surfacing from the root README, add a link using the
  published URL pattern `https://khsacc.github.io/FluoRaPressee/docs/<path>` — see
  `README_ja.md`'s links to `docs/hardware-config`, `docs/api`, `docs/data-formats` for the pattern.
  Not required for every page — only when the README currently has inline detail that the new page
  supersedes (as happened when the setup-wizard details moved into `hardware-config.md`) or otherwise
  clearly warrants a pointer.

## Verify (mirrors CI exactly — do this before considering the page done)

```bash
cd docs-site
npm run typecheck   # tsc — same as test-docs.yml
npm run build       # catches onBrokenLinks:'throw' failures — same as test-docs.yml and deploy-docs.yml
```
Then preview locally and actually look at the rendered page and sidebar placement:
```bash
npm start           # http://localhost:3000/FluoRaPressee/
```
Confirm: the page appears in the sidebar at the intended position/nesting, the heading/title render
correctly, any images load, and every cross-link you added actually navigates (a link that resolves
at build time can still point at the wrong page).

Once merged to `master` (the repo's actual default branch — `deploy-docs.yml` triggers on push to
`master`), GitHub Pages publishing is automatic; no manual deploy step is needed.
